## Goal

Move the Busytok macOS updater from "silent auto-apply on launch" to a mainstream,
user-controlled update model, and add a safe downgrade escape hatch. Four changes:

1. **Periodic polling** in the GUI (check on launch + on a low-frequency timer while
   running), instead of a single launch-time check.
2. **A badge update button** in the titlebar (hidden until an update is detected; one
   click downloads + installs + relaunches), replacing the current silent auto-apply.
   Release notes are surfaced via tooltip. The Settings-page manual check is kept but
   rewired to the shared state.
3. **Manual downgrade** from a version-history manifest, reusing the existing updater
   pipeline. No crash auto-detection (deliberate — see Decisions).
4. **Windows**: explicitly documented as "auto-update not supported" (README + Settings
   note). No CI work this round.

## Background — current updater (as of 2026-06-23)

- Tauri 2 built-in updater (`tauri-plugin-updater = "=2.10.1"`), four-step wiring all
  present (Cargo dep, npm dep, `lib.rs` plugin registration, `capabilities` `updater:default`).
- Endpoint: `https://github.com/BalianWang/busytok/releases/latest/download/latest.json`
  (GitHub Releases as free CDN; `latest.json` generated by repo-owned
  `scripts/generate_updater_manifest.sh`).
- Signature: minisign/ed25519 (pubkey embedded in `tauri.conf.json`), layered on top of
  Apple Developer ID signing + notarization.
- Trigger: `apps/gui/src/main.tsx` fires `initUpdaterAutoCheck()` once at module load →
  `checkAndApplyUpdate()` in `apps/gui/src/lib/updaterClient.ts`, which does
  `check() → downloadAndInstall() → relaunch()` **with no user prompt**. No periodic
  timer, no push, no service-side check.
- `apps/gui/src/hooks/useUpdater.ts` exposes a manual `checkNow` used by a Settings-page
  button; it also calls `checkAndApplyUpdate` (i.e. also auto-applies).
- `latest.json` `notes` field is fetched by the plugin but **never surfaced** to the user.
- macOS universal only; `latest.json` has no `windows` platform entry.
- The background `busytok-service` (LaunchAgent) has **no** update logic and no HTTP
  client dependency — updates are GUI-side only.
- The Rust module `apps/gui/src-tauri/src/updater.rs` is **only a logging stub**
  (`init_updater_logging()` emits one tracing line). All check/download/install runs
  through the plugin's JS API; the Rust side drives nothing today. This matters for §5
  (downgrade): the command is **net-new Rust** but drives the plugin's *public* `UpdaterExt`
  API — it does not hand-roll download/verify/install logic.

## Decisions (confirmed with owner)

- **D1 — Split check from apply.** The launch path stops auto-applying. `checkForUpdate()`
  only detects and sets state; `applyUpdate()` (download + install + relaunch) runs only
  when the user clicks the badge button. Aligns with Sparkle / electron-updater defaults.
- **D2 — Rollback = manual downgrade entry (not auto-restore).** Publish a version-history
  manifest; Settings page lists recent versions; user picks one and reinstalls via the
  existing updater pipeline. **No crash auto-detection / health-check launcher.** Rationale:
  reuses proven pipeline, avoids fragile macOS crash detection, gives users an escape hatch.
  Full auto-restore is explicitly out of scope.
- **D3 — Windows = document as unsupported.** README + Settings note. No Windows CI build
  job this round (none exists; Windows support overall is still in-flight). Wiring Windows
  auto-update is deferred until Windows is a shipping target with a build pipeline.
- **D4 — Cluster badge + toolbar at top-right (corrected).** Wrap the global
  `UpdateBadgeButton` and the per-page toolbar slot in a single right-aligned group
  `.desktop-titlebar__actions`. Because the parent is already `space-between` and the
  toolbar already has `margin-left: auto`, the only CSS change is: **remove**
  `margin-left: auto` from `.desktop-titlebar__toolbar` and put it (once) on the new
  `.desktop-titlebar__actions` wrapper. One `margin-left: auto`, no double-push, no
  confusion for future maintainers.
- **D5 — One click = direct install.** Clicking the badge button calls `applyUpdate()`
  immediately (download + install + relaunch). Release notes are shown via button tooltip
  (`update.body`), not a blocking confirm dialog. Owner explicitly wants direct install.
  Direct install does **not** mean silent failure: download progress is surfaced (D9) and a
  failed relaunch falls back to a manual-restart prompt (D12).
- **D6 — Polling stays frontend-driven.** The interval lives in the React layer
  (`UpdaterProvider`), consistent with the existing frontend-driven check. A Rust tokio
  task is rejected as over-engineering for this app.
- **D7 — Version-history source = `versions.json` artifact (not runtime GitHub API).**
  CI generates a small manifest of the last N releases; the app fetches it. Deterministic,
  no runtime API dependency / rate-limit / shape drift — matches the repo-owned-manifest
  philosophy of `generate_updater_manifest.sh`.
- **D8 — The `Update` handle is a managed resource.** Tauri's JS `Update extends Resource`
  (server-side handle + `close()`). The provider holds the live `Update` in a `useRef`
  (not React state) and **closes it before swapping** on every subsequent available result,
  on unmount, and on terminal errors that discard it. Public state carries only metadata.
- **D9 — Download progress is surfaced, cheaply.** `downloadAndInstall(onEvent)` yields
  `Started { contentLength? }` / `Progress { chunkLength }` / `Finished`. The provider
  accumulates `percent` and exposes it in the `downloading` state. The tiny badge shows
  `Update <n>%` (label) with the version+notes in the tooltip; the Settings surface shows
  a percentage + message. This is near-free given the callback already exists.
- **D10 — Bounded timeouts on check and download.** `CheckOptions.timeout` and
  `DownloadOptions.timeout` both exist. Defaults: **20s** check, **120s** download (a
  worst-case slow link for the ~10–30 MB `.app.tar.gz`); both configurable via Settings.
  Prevents a hung check from occupying the UI when GitHub is unreachable.
- **D11 — Manual check resets the poll timer.** A user-initiated `checkNow()` clears and
  restarts the 12h interval so the next automatic check is 12h from the manual one, not a
  redundant re-check minutes later.
- **D12 — Relaunch failure is a first-class state.** `relaunch()` (from
  `@tauri-apps/plugin-process`) can reject or race with exit (e.g. sandbox/permission
  edge cases). `installed-pending-restart` therefore has a failure branch
  `installed-needs-manual-restart`: "Update installed — please restart Busytok manually,"
  with no spinner hang. A relaunch failure is also telemetered.

## Design

### §1 Shared update state — `UpdaterProvider`

A new React context provider owns the single source of truth for update state. It is
mounted in **`apps/gui/src/App.tsx`** (see §2), alongside the existing
`EventSubscriptionProvider` / `PageToolbarProvider`. Both the AppShell badge button and the
Settings manual-check button consume it.

**Where the live `Update` lives (D8):** in a `useRef<Update | null>`, never in React state.
React state carries only the serializable `status` union — whose `available` variant holds the
metadata `{ version, notes, date }` (the live `Update` object itself is not in state). The ref
is closed before any swap and on unmount.

State machine (states, transitions, resource + retry rules):

```
idle ──(check)──▶ checking ──▶ up-to-date
                          ├──▶ available { version, notes, date }   (updateRef holds Update)
                          └──▶ error { message, retryable }

available ──(apply)──▶ downloading { percent? } ──▶ installed-pending-restart ──▶ relaunch → app exits
available ──(recheck: interval / focus / manual)──▶ [close() old Update] ──▶ checking ──▶ ...
downloading ──(download error)──▶ available   (Update still valid → user can retry apply)
downloading ──(downloadAndInstall ok, relaunch ok)──▶ installed-pending-restart → relaunch
downloading ──(downloadAndInstall ok, relaunch FAILS)──▶ installed-needs-manual-restart (D12)
error ──(retry)──▶ checking | available | downloading   (whichever the failed step was)
```

Transition clarifications (audit items #3, #12):

- A download/install error returns to **`available`**, not `idle` — the `Update` object is
  still held and valid, so the user can retry the *apply* without a fresh check. Only a
  terminal error that discards the Update (e.g. Update was closed) routes to `idle`/`error`.
- After `installed-pending-restart`, if `relaunch()` rejects or the app does not exit, the
  state becomes **`installed-needs-manual-restart`** with a manual-restart message (D12).

The provider exposes: `status` — the discriminated union above, where each variant carries
its own payload (`available` → version/notes/date, `downloading` → `percent`, `error` →
message) — plus actions `checkNow()` and `applyNow()`.

**`useUpdater()` is the single consumer hook** (audit #6). It reads the context and returns
`{ status, checkNow, applyNow }`, where `status` is the discriminated union above — the badge
reads `percent` off the `downloading` variant and version/notes off the `available` variant,
so no separate progress/available field is needed. `AppShell`/`UpdateBadgeButton` and
`SettingsPage` both call `useUpdater()`; there is no second, parallel API. The old
self-contained `useUpdater` (local `checkAndApplyUpdate` + local status) is removed.

`apps/gui/src/lib/updaterClient.ts` is refactored from one `checkAndApplyUpdate` into:

- `checkForUpdate(): Promise<{ kind: 'up-to-date' } | { kind: 'available'; version; notes; date; update } | { kind: 'error'; message }>`
  — passes `CheckOptions { timeout }` (D10).
- `applyUpdate(update, onProgress?): Promise<{ kind: 'updated'; version } | { kind: 'error'; message } | { kind: 'needs-manual-restart' }>`
  — calls `update.downloadAndInstall(onProgress, { timeout })` then `relaunch()`; maps a
  `relaunch()` rejection to `needs-manual-restart` (D12) so the UI never hangs.

### §2 Polling

The provider is mounted in **`App.tsx`** (audit #4) — NOT `main.tsx`. This is both more
correct (the panel window must not poll) and matches where `EventSubscriptionProvider` /
`PageToolbarProvider` already live. `main.tsx`'s only updater change is **removing the
`initUpdaterAutoCheck` import + call** (`if (!promptPaletteWindow)` block).

Driven inside `UpdaterProvider`:

- **On mount** (replaces `main.tsx`'s `initUpdaterAutoCheck`): one `checkForUpdate()`.
- **Interval**: re-check every **12h** while the app is open (configurable via a Settings
  field; default 12h). `setInterval`, cleared on unmount; StrictMode-safe (the existing
  module-level latch pattern is unnecessary inside a provider whose `useEffect` cleanup is
  the dedup, but the effect must be written to tolerate the dev double-invoke: the interval
  is created in the effect and torn down in cleanup, so StrictMode's mount-unmount-mount
  leaves exactly one interval).
- **On focus**: use Tauri's `getCurrentWindow().onFocusChanged()` (audit #7) rather than
  `visibilitychange`. Re-check only if the last check was >1h ago (avoids re-checking on
  every minor focus blip). More precise than `visibilitychange` for distinguishing
  app-switch returns from Dock restores.
- **Manual check resets the timer (D11)**: `checkNow()` clears and restarts the 12h interval.
- **Timeouts (D10)**: every `checkForUpdate()` / `applyUpdate()` call passes the configured
  check/download timeouts so an unreachable GitHub can't stall the UI.

Polling frequency is conservative (update checks are not time-critical for a local audit
tool) and only runs while the GUI is open — the background service is unchanged.

### §3 The badge update button

New component `apps/gui/src/components/desktop/UpdateBadgeButton.tsx`, rendered globally in
`AppShell.tsx`, consuming `useUpdater()`:

- **Visibility**: hidden unless `status` is `available` / `downloading` /
  `installed-needs-manual-restart` / `error` (once triggered). No chrome when `up-to-date`.
- **Available state**: compact button (icon + dot) whose `title`/tooltip shows
  `v<version> available` + the release notes body.
- **Click** → `applyNow()` → `downloading` showing `Update <percent>%` (D9; tooltip keeps
  version+notes) → `installed-pending-restart` ("Updated to vX — restarting…") → relaunch.
- **Error state**: returns to `available` for retry (audit #3) and shows the failure message.
- **Relaunch failure (D12)**: shows `installed-needs-manual-restart` — "Updated to vX —
  please restart Busytok manually." No spinner hang.
- **Resource safety (D8)**: because the provider owns `updateRef` and closes before swap,
  the badge never holds a stale `Update`; a re-check while available replaces the held
  handle cleanly.
- **Placement (D4, corrected)**: `AppShell.tsx` wraps `{updateBadgeButton}{toolbarContext.toolbar}`
  in a new right-aligned container `.desktop-titlebar__actions { margin-left: auto;
  display: flex; gap; align-items: center }`, and the existing `margin-left: auto` is
  **removed** from `.desktop-titlebar__toolbar`. The badge sits immediately left of the
  per-page refresh button, both pinned top-right as one group.

### §5 Manual downgrade — install path (R1 resolved 2026-06-23)

R1 has been **verified against the installed crate source**
(`tauri-plugin-updater-2.10.1/src/updater.rs`): the plugin's public Rust API installs a
user-chosen older version cleanly. **No manual fetch+verify+replace; the browser-DMG fallback
is demoted to a last resort.**

**Mechanism.** The downgrade command uses `UpdaterExt::updater_builder()` — the builder is
seeded with the already-embedded `pubkey` + config from `tauri.conf.json` — then overrides
two things per call:

1. `.endpoints(vec![<per-tag manifest url>])` (`updater.rs:197`) — points the check at the
   chosen older version's manifest instead of the configured "latest" endpoint.
2. `.version_comparator(|_current, _remote| true)` (`updater.rs:184`) — forces
   `should_update = true` (`updater.rs:530-533`). Without this `check()` returns `None` for an
   older version (default comparator is `release.version > current`). **This is the real
   downgrade enabler** — distinct from the JS `CheckOptions.allowDowngrades`, which only
   relaxes the comparison against the single configured endpoint and cannot target an
   arbitrary version.

Then `.build()?.check().await?` resolves an `Update`, and `Update::download_and_install(...)`
(`updater.rs:723`) downloads, verifies the minisign/ed25519 signature against the embedded
pubkey (`updater.rs:712`), and installs via the **same macOS pipeline as regular updates**
(`install_inner`, `updater.rs:1217` — extract `.app.tar.gz`, back up the current app, swap in
place; AppleScript `with administrator privileges` when `/Applications` needs elevation).

**`check()` is endpoint-driven, not url+sig-driven** — it fetches a `RemoteRelease`-shaped
JSON manifest (`{ version, notes, pub_date, url, signature }`), so the downgrade target must
be addressable as a per-tag manifest. It is.

**CI fact (no new manifest-upload work).** `release.yml:154-183` already uploads each tag's
own `latest.json` as a release asset, so `releases/download/<tag>/latest.json` resolves for
every historical release this repo's pipeline has produced. Downgrade reuses these.

**`versions.json` (D7) is therefore a lightweight index**, not a payload manifest:
`{ version, date, notes, manifest_url }` per entry, last **N=5**, published to
`releases/latest/download/versions.json`. **No `signature` field** (audit #10 resolved) — the
signature lives inside each per-tag `latest.json` that `check()` fetches and verifies. The
frontend fetches `versions.json` to render the list; "Reinstall" passes the entry's
`manifest_url` to the Rust command.

**New Rust command** (net-new — `updater.rs` itself stays the logging stub; this lives in
`commands.rs`, registered in `lib.rs`):

```rust
#[tauri::command]
async fn install_version(app: AppHandle, manifest_url: String) -> Result<InstallOutcome, String> {
    use tauri_plugin_updater::UpdaterExt;
    let update = app
        .updater_builder()                                  // carries embedded pubkey + config
        .endpoints(vec![manifest_url.parse().map_err(|e| e.to_string())?])
        .map_err(|e| e.to_string())?                        // per-tag latest.json
        .version_comparator(|_current, _remote| true)       // allow the older version
        .build()
        .map_err(|e| e.to_string())?
        .check()
        .await
        .map_err(|e| e.to_string())?
        .ok_or("no update resolved".to_string())?;          // comparator=true ⇒ Some
    update
        // on_chunk: (bytes in this chunk, total content-length or None when header absent)
        .download_and_install(|_chunk_len: usize, _content_length: Option<u64>| {}, || {})
        .await
        .map_err(|e| e.to_string())?;
    // relaunch via app.request_restart() — Tauri core, no extra dep (the process plugin's
    // JS relaunch() is the existing frontend path, unchanged). Reuse the D12 manual-restart
    // fallback if the restart doesn't take.
    Ok(InstallOutcome { /* version, needs_manual_restart */ })
}
```

**Implementation notes (verified constraints):**
- Comparator closure must be `Fn(Version, RemoteRelease) -> bool + Send + Sync + 'static`;
  `|_, _| true` captures nothing and satisfies this. This call intentionally overrides any
  global comparator `updater_builder()` seeds from `UpdaterState` (lib.rs) — `None` in this
  app today, but the override is what guarantees the older version is accepted.
- Key consistency: any release signed with this repo's keypair verifies against the embedded
  pubkey. Pre-updater tags lacking `latest.json` simply aren't indexed (R2 covers the short
  early list).
- macOS elevation UX (AppleScript password prompt when needed) is identical to regular
  updates — not a downgrade-specific burden.
- Rust-level progress is available via `download_and_install`'s `on_chunk(usize, Option<u64>)`;
  optional Tauri-event streaming to the Settings UI (non-blocking).
- New permission `allow-install-version` in `capabilities/default.json` (existing `allow-*`
  convention).

**Last-resort fallback only** (not the main path): if a future plugin change ever breaks the
`UpdaterBuilder` override surface, downgrade can open the chosen release's DMG in the browser
for a manual drag-install; in that mode `versions.json` would carry a `dmg_url` instead of
`manifest_url`. Not needed for 2.10.1.

### §8 Transition safety

Changing the launch path from silent-auto-apply to detect-only is a behavioral change.
Existing users on the current build will be silently auto-updated **one more time** (the
old code still auto-applies) to a build containing the new flow; all subsequent updates
then use the badge flow. Self-healing — no data migration, no feature flag needed.

Relaunch reliability (audit #12): because the badge/apply path depends on `relaunch()`,
D12 ensures a failed relaunch surfaces a manual-restart message instead of leaving the user
on a stuck spinner; combined with D10 timeouts, no update operation can hang the UI.

## Phasing (for the implementation plan)

- **P1** — `UpdaterProvider` (with resource lifecycle + timeouts) + refactored client +
  periodic polling + badge button (with progress + relaunch-failure handling) + top-right
  clustered layout (single `margin-left:auto`) + Settings rewire. (The core UX change;
  shippable alone.)
- **P2** — Manual downgrade: `versions.json` index + CI step + `install_version` Rust
  command (R1-resolved API) + Settings version-history panel. (R1 resolved 2026-06-23;
  unblocked.)
- **P3** — Windows "not supported" docs (README + Settings, gated on `isMacPlatform()`).
  Trivial; can ride with P1.

## Risks & open questions

- **R1 (low, resolved 2026-06-23) — Downgrade install API.** Verified against crate source
  (§5): the plugin's public `UpdaterBuilder` (`.endpoints()` + `.version_comparator(|_,_|true)`)
  installs a chosen older version via the same proven macOS pipeline. `check()` is
  endpoint-driven, so the target must be a per-tag manifest (`releases/download/<tag>/latest.json`)
  — already uploaded by `release.yml`. `signature` is NOT in `versions.json` (it lives in each
  per-tag manifest). Residual constraints: the comparator closure must be `Send+Sync+'static`;
  only releases signed with this repo's keypair verify; macOS elevation UX equals regular
  updates. Browser-DMG is a last-resort fallback only, not needed for 2.10.1.
- **R2 (low) — `versions.json` for the very first releases.** Until N releases accumulate,
  the list is short; that's fine (downgrade to whatever exists).
- **R3 (low) — Top-right layout shift.** Wrapping the toolbar in `.desktop-titlebar__actions`
  is a global visual change; verify each page's toolbar still looks correct
  (Overview/Usage/Prompt Palette/Activity/Settings/BreakdownLedger). The corrected D4 makes
  this *less* invasive than the prior draft (one `margin-left:auto`, toolbar already
  right-aligned).
- **R4 (low) — Relaunch reliability.** `relaunch()` may reject on sandbox/permission edges;
  mitigated by D12 (manual-restart state + telemetry) and D10 timeouts. No silent hang.
- **Q1** — Polling interval default 12h acceptable, or shorter? (Defaulted to 12h;
  configurable.)
- **Q2** — Badge button visual: icon+dot only, or "Update" label? Resolved for the progress
  case ("Update <n>%"); the non-progress available-state visual (icon+dot vs label) is left
  to implementation polish — tooltip carries version+notes either way.

## Out of scope

- Differential/delta updates (full replace retained).
- Staged/canary rollout and release channels.
- Moving update checks into the background `busytok-service`.
- Release-notes confirm dialog (D5 — direct install chosen).
